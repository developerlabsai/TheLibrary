Quickstart Guide

Get up and running in under 5 minutes. This guide walks through obtaining credentials, making your first API call, and handling the response.

Prerequisites

  • An active account with API access enabled
  • Your client_id and client_secret from the developer portal
  • A tool for making HTTP requests (curl, Postman, or your language's HTTP client)

Step 1: Obtain an Access Token

# Exchange client credentials for an access token curl -X POST https://api.example.com/v2/auth/token \ -H "Content-Type: application/json" \ -d '{ "client_id": "your_client_id", "client_secret": "your_client_secret", "grant_type": "client_credentials" }'

Step 2: Make Your First Request

# Fetch your account details curl https://api.example.com/v2/account \ -H "Authorization: Bearer YOUR_ACCESS_TOKEN"

Step 3: Parse the Response

{ "id": "acc_8x7k2m9n", "name": "Acme Corp", "plan": "professional", "created_at": "2026-01-15T10:30:00Z", "rate_limit": { "requests_per_minute": 1000, "remaining": 998 } }
Success
If you received a 200 response with your account details, you're all set. Continue to the Authentication section for production-ready token management.

04

Authentication

The API uses OAuth 2.0 with the Client Credentials grant type. All requests must include a valid Bearer token in the Authorization header.

Token Lifecycle

  1. Request a token by sending your client_id and client_secret to the /auth/token endpoint.
  2. Include the token in the Authorization: Bearer <token> header for all API calls.
  3. Refresh the token before expiry. Tokens are valid for 3600 seconds (1 hour).
  4. Handle 401 errors by requesting a new token and retrying the failed request.
Security Notice
Never expose your client_secret in client-side code, version control, or logs. Use environment variables or a secrets manager. Rotate credentials immediately if compromised.

API Endpoints

All endpoints follow REST conventions. Request and response bodies use JSON. Timestamps are ISO 8601. IDs are prefixed strings (e.g., usr_abc123).

GET /v2/contacts List all contacts

Returns a paginated list of contacts. Supports filtering by status, created_after, and tags.

ParameterTypeRequiredDescription
pageintegerNoPage number (default: 1)
per_pageintegerNoResults per page (default: 25, max: 100)
statusstringNoFilter: active, inactive, archived
created_afterISO 8601NoFilter by creation date
POST /v2/contacts Create a contact

Creates a new contact. Returns the created resource with a unique ID.

{ "email": "jane@acme.com", "first_name": "Jane", "last_name": "Smith", "company": "Acme Corp", "tags": ["enterprise", "inbound"] }
PUT /v2/contacts/:id Update a contact

Updates an existing contact. Only provided fields are modified (partial update supported).

DELETE /v2/contacts/:id Delete a contact

Permanently deletes a contact. This action is irreversible. Requires admin scope.

Error Handling

The API uses standard HTTP status codes and returns structured error objects. Always check the error.code field for programmatic error handling.

Error Response Format

{ "error": { "code": "validation_error", "message": "The email field is required", "details": [ { "field": "email", "rule": "required", "message": "Email address is required" } ], "request_id": "req_9f8e7d6c" } }

HTTP Status Codes

CodeMeaningWhen It Happens
200OKSuccessful GET, PUT requests
201CreatedSuccessful POST (resource created)
204No ContentSuccessful DELETE
400Bad RequestValidation error, malformed JSON
401UnauthorizedMissing or expired token
403ForbiddenInsufficient scope/permissions
404Not FoundResource doesn't exist
429Rate LimitedToo many requests (check Retry-After header)
500Server ErrorInternal error (report with request_id)
Rate Limiting
When you receive a 429, always respect the Retry-After header. Implement exponential backoff for retries. Persistent rate limit violations may result in temporary API key suspension.

Deployment Guide

This section covers deployment configuration for self-hosted integrations. Follow these steps to configure your environment for production use.

Environment Variables

# Required configuration API_BASE_URL=https://api.example.com/v2 API_CLIENT_ID=your_client_id API_CLIENT_SECRET=your_client_secret API_TIMEOUT_MS=30000 # Optional configuration API_RETRY_COUNT=3 API_RETRY_DELAY_MS=1000 LOG_LEVEL=info WEBHOOK_SECRET=whsec_your_webhook_signing_secret

Deployment Checklist

  1. Configure credentials — Set environment variables in your secrets manager (not in code)
  2. Set up webhook endpoint — Deploy a publicly accessible HTTPS endpoint for webhooks
  3. Validate connectivity — Run the health check endpoint GET /v2/health
  4. Enable monitoring — Set up alerting for 4xx/5xx error rates and response latency
  5. Test failover — Verify retry logic handles transient errors gracefully
